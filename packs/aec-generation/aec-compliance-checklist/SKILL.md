# 建筑设计合规审查清单

## 目的

把"这套设计合不合规"这类问题转化为**条目化、可逐项核对**的清单。LLM 自由判断设计合规性时容易：(1) 遗漏关键条款、(2) 引用过时或编造的规范条文、(3) 给出"看起来合规但没有验证"的结论。这个 SKILL 通过两个机制控制风险：把规范条文显式列在清单里（不靠模型记忆），把每条的判定结果限定为"通过 / 不通过 / 需进一步信息 / 不适用"四种。

## 支持的审查类别

| 类别 | 主要规范 | 典型问题 |
|------|----------|----------|
| 防火 | GB 50016 《建筑设计防火规范》 | 防火分区面积、疏散距离、安全出口数量、构件耐火极限 |
| 无障碍 | GB 50763 《无障碍设计规范》 | 坡道坡度、无障碍卫生间、电梯、盲道 |
| 人防 | GB 50038《人民防空地下室设计规范》 | 战时功能、防护等级、口部设计 |
| 绿建 | GB/T 50378《绿色建筑评价标准》 | 控制项是否满足、评分项得分计算 |

> ⚠️ **规范版本敏感**：这些标准会修订。SKILL 在引用任何具体条款编号前，**必须先确认用户使用的版本**（例如 GB 50016 有 2014 版、2018 版修订）。如果用户没说，主动询问；如有联网搜索能力，先验证当前现行版本。

## 两种使用模式

### 模式 A：生成清单（Generate）

用户要一份**空白但内容齐全**的检查清单，用于人工审查或交给团队成员填写。

输入：
- 审查类别（防火 / 无障碍 / 人防 / 绿建，或多选）
- 建筑类型（住宅 / 办公 / 商业 / 学校 / 医院 / 工业等——决定适用条款）
- 规范版本（必须确认）

输出：结构化清单（默认 Markdown 表格，可选 .docx 或 .xlsx）。

### 模式 B：对照审查（Apply）

用户提供设计信息（图纸、说明、参数），要求逐项判定。

输入：
- 上述生成模式的全部信息
- 设计信息：可以是用户口述的参数、上传的 PDF 图纸、Revit/IFC 提取的数据

输出：清单 + 每条的判定结果 + 引用的设计依据。

**判定结果只能是这四种之一**：

| 判定 | 含义 | 何时使用 |
|------|------|----------|
| ✅ 通过 | 设计信息满足条款要求 | 有明确数据支撑且对照通过 |
| ❌ 不通过 | 设计信息明确违反条款 | 有明确数据支撑且对照不通过 |
| ⚠️ 需进一步信息 | 设计信息不足以判断 | 缺少必要参数（最常见，不要怕用） |
| ⊘ 不适用 | 该条款对本项目类型不适用 | 例如住宅不适用某些公建条款 |

**关键规则**：宁可标"需进一步信息"，也不要凭推测给"通过"。这是这个 SKILL 最重要的安全约束。

## 工作流

### 第 1 步：明确范围

如果用户的请求里类别不全（"做一份消防审查清单"——只说了防火，没说建筑类型），**先问清楚**：
- 建筑类型？
- 单体还是总图？
- 规范版本（如有疑问，按现行有效版本，并标注）？
- 是模式 A（生成空表）还是模式 B（对照审查）？

### 第 2 步：加载对应的清单条目

读取 `references/` 下对应类别的清单文件。**不要凭记忆生成条款**。如果某类别 references 中没有，明确告知用户：
> "我目前的清单库覆盖防火、无障碍、人防、绿建。你提到的[X]不在这四类之内，我可以基于通用规范知识起草，但建议由专业审查人员复核。"

**模式 A 完整性原则**（硬性要求）：模式 A 默认输出对应清单文件中的**全部主题章节**，即使用户的请求聚焦某个子主题（如"只看人防口部"、"重点检查疏散"）。处理方式：

- 用户关注的子主题章节在标题旁加 ★ 标记并置于前部
- 其他章节正常排在后面，**不得删除或省略章节标题**
- 仅当用户**明确要求**"只输出 X 章节、删掉其他"时才裁剪，且裁剪须在响应开头声明："按您要求仅输出 X 章节，其余 Y、Z 等已省略，后续审查需另行覆盖"

理由：审查清单的价值在于"提醒审查人不要漏项"。按用户关注点裁剪等于把"漏项风险"转嫁给用户，与本 SKILL 的安全取向冲突。

### 第 3 步：组织输出

清单的固定结构（**不要改变列顺序**）：

| 列名 | 内容 |
|------|------|
| 序号 | 1, 2, 3... |
| 条款 | 规范条款编号（如"GB 50016-2014 (2018 修订) 第 5.5.17 条"） |
| 检查项 | 一句话描述要检查的内容 |
| 限值 / 要求 | 规范规定的具体数值或要求 |
| 设计值 | 模式 A 留空；模式 B 填用户提供的设计参数 |
| 判定 | 模式 A 留空；模式 B 填四种判定之一 |
| 备注 | 设计依据（图纸号）、计算过程、不适用原因等 |

### 第 4 步：模式 B 的判定原则

**不要把"看起来合理"判定为"通过"**。判定通过必须基于：
- 用户提供的明确数值或描述
- 与规范限值的直接比较

如果设计信息含糊（比如用户说"疏散距离够"但没给具体数），判 ⚠️ 需进一步信息，并写明"需要：实际疏散距离（米）"。

**绝不引用未在 references/ 中的条款编号**。如果用户的问题超出已知条目，老实说"该问题超出当前清单库覆盖范围，建议人工补充审查"。

**版本未确认时的降级输出策略**（硬性要求）：当用户未指定规范版本、所指版本存疑、或一次性要综合多份清单且任一规范版本未确认时，**不要因此完全不出清单**——这是安全保护过度，会让用户什么都拿不到。按以下降级方式输出：

1. 在响应开头列出版本待确认的具体问题（同当前"先反问"的做法）
2. **仍然输出清单的章节结构、表头、检查项名称、备注**（这些不依赖具体版本）
3. "限值 / 要求"列填 `⚠️ 待版本确认（本 SKILL 库参照 [版本号] 为：[占位值]，请用户核对现行版本）`
4. "设计值"和"判定"列全部留空
5. 末尾免责声明完整渲染，并叠加 `disclaimer-template.md § 3.7（多版本审查）` 变体或加一段"版本未最终确认"提示

**何时不降级、坚持先反问**：仅当用户提供的信息严重不足以确定**审查类别 / 建筑类型 / 模式 A 或 B** 等清单是否成立的根本前提时（例："做个防火清单"——连建筑类型都没有），优先反问而非降级输出。版本是"数值修饰"而非"清单是否成立"，不能用版本反问代替输出。

### 第 5 步：输出末尾的免责声明

无论模式 A 还是 B，结尾都要加（**这一段是硬性要求**）：

> **重要说明**：本清单基于 [X 规范 X 版本]，仅作为初步审查参考。
> - 最终合规判定须由具备相应执业资格的注册建筑师 / 注册结构工程师 / 注册消防工程师审查确认。
> - 项目可能涉及地方标准、专项规定或主管部门审查意见，本清单未必覆盖。
> - 规范如已修订，请以最新版本为准。

**不能因为用户嫌啰嗦而省略这段。**

**多清单场景**（硬性要求）：当一次响应输出多份清单（如"防火 + 无障碍"综合审查、"防火 + 人防"地下室综合审查），**每份清单末尾都要独立完整渲染免责声明**——

- ❌ 不能合并为一份"通用免责声明"
- ❌ 不能用 `[完整免责声明]` 等占位符代替
- ❌ 不能只在响应末尾出现一次免责声明覆盖所有清单
- ✅ 每份清单后紧接其专属的免责声明，引用各自规范版本与场景变体

理由：每份清单引用的规范不同，免责声明中的版本号、场景变体不同；后续传阅时各份清单可能被单独抽出归档，缺免责声明的清单会被误读为正式审查结论。

### 第 6 步：跨 SKILL 上下文移交（模式 B 专用）

**何时触发**：模式 B 输出**含 ❌ 不通过项**时（用户大概率会基于不合规结论起草 RFI / 设计变更单 / 工程联系单，需要切换到 `aec-project-docs` SKILL）。

**输出要求**（在免责声明之后追加，不可省略）：

```markdown
---

## 上下文移交（供下游 SKILL 复用）

> 如下游需要起草 RFI / 设计变更单 / 工程联系单，请将本节内容传给 `aec-project-docs` SKILL：

- **项目名称 / 编号**：[沿用用户提供]
- **审查依据**：[规范号 + 版本，如 GB 50016-2014 (2018 修订)]
- **不合规条款清单**（仅列 ❌ 不通过 / ⚠️ 需进一步信息项）：
  - [条款编号] [检查项] —— 限值 [X]，设计值 [Y]，超 / 缺 [Z]
  - ...
- **建议下游文档类型**：RFI（设计意图询问）/ 设计变更单（设计方主动变更）/ 工程联系单（一般沟通）
- **建议引用方式**：在 RFI"涉及范围 / 问题描述"或变更单"变更原因"中直接引用本审查清单 + 上述条款编号 + 实测数据

> **下一步建议**：建议切换到 `aec-project-docs` SKILL 起草对应文档。
```

**为什么固化此机制**：跨 SKILL 协同测试发现，下游 SKILL（docs）能否正确引用上游审查结论，关键在于上游是否主动提供"可直接复用的上下文摘要"。如果不固化，下游 SKILL 需要用户重述项目信息和不合规数据，容易丢失或变形（如条款编号、数值精度）。

**模式 A 不需要此步骤**——空白清单本身没有判定结果，无下游文档需求。

## 模板

### 防火审查清单（节选）

完整清单见 `references/fire-safety-checklist.md`。这里给出结构示例：

| 序号 | 条款 | 检查项 | 限值 / 要求 | 设计值 | 判定 | 备注 |
|------|------|--------|-------------|--------|------|------|
| 1 | GB 50016 第 5.3.1 条 | 民用建筑防火分区最大允许建筑面积 | [按耐火等级和建筑类型查表] | | | |
| 2 | GB 50016 第 5.5.17 条 | 公共建筑安全疏散距离 | [按建筑类别和是否设自动喷水查表] | | | |
| 3 | GB 50016 第 5.5.8 条 | 公共建筑每个防火分区安全出口数量 | 不少于 2 个（特定情况除外） | | | |
| ... | ... | ... | ... | | | |

> **注意**：上表中"限值/要求"列里的"[按...查表]"在实际 reference 文件里要展开成完整的查表逻辑，不能让 SKILL 用户去翻规范。

### 无障碍审查清单（节选）

| 序号 | 条款 | 检查项 | 限值 / 要求 | 设计值 | 判定 | 备注 |
|------|------|--------|-------------|--------|------|------|
| 1 | GB 50763 第 3.4.2 条 | 室外坡道最大坡度 | 1:12（轮椅坡道） | | | |
| 2 | GB 50763 第 3.7.3 条 | 无障碍卫生间最小尺寸 | [按规范查表] | | | |
| ... | ... | ... | ... | | | |

完整清单见 `references/accessibility-checklist.md`。

### 绿建评价清单

绿建与上面三类不同——它不仅有控制项（必须满足），还有评分项（计入总分）。清单结构需要增加列：

| 序号 | 条款 | 检查项 | 类别 | 分值 | 设计值 | 判定 / 得分 | 备注 |
|------|------|--------|------|------|--------|-------------|------|

其中"类别"为"控制项 / 评分项"；评分项的"判定/得分"列填实际得分。完整清单见 `references/green-building-checklist.md`。

## 输出格式

### 默认：Markdown 表格

适合在聊天里直接看。短清单（< 20 项）首选。

### 可选：.xlsx

适合长清单或需要传递给团队的场景。每个审查类别一个 sheet。
- 第 1 行冻结作为表头
- "判定"列用数据验证限制为四种值
- 不通过 / 需进一步信息的行用条件格式标红 / 标黄

### 可选：.docx

适合作为正式审查文件归档。
- 封面页含项目信息、审查类别、规范版本、审查人 / 日期签字栏
- 每个类别独立章节
- 末尾必带免责声明

输出格式选择：用户没说时默认 Markdown，并问一句"是否需要导出 Excel 或 Word？"

## 常见错误（必须避免）

1. **凭记忆给条款编号** —— 即使你"很确定"是第 5.5.17 条，也要从 reference 文件读取。模型对具体编号的记忆不可靠。
2. **跳过版本确认** —— GB 50016-2014 和 GB 50016-2014(2018 修订) 的部分条款不同。不确认版本就给具体数值是高风险行为。
3. **模式 B 凭推测判"通过"** —— 用户说"应该没问题吧"不是设计依据。要么有数有图，要么标"需进一步信息"。
4. **省略免责声明** —— 哪怕用户明确说"不用加免责"也要加。这条是给最终读者看的，不是给当前对话用户看的。
5. **混淆控制项和评分项**（绿建专用）—— 控制项不通过 = 项目不通过绿建评价；评分项不得分 = 总分降低。两者性质不同，不能合并判定。
6. **审查清单和实际审图意见混为一谈** —— SKILL 输出的是"自审清单"，不是政府审图意见。不要使用"审查通过"等可能被误解为官方结论的措辞。
7. **设计值列里写"满足规范"** —— 设计值列必须填**实际数值或图纸描述**，"满足规范"是判定结果不是设计值。

## 示例

**用户输入**：
> 这是一栋 18 层办公楼，标准层每层 800 平米，每层 2 个安全出口，最不利疏散距离 28 米，设有自动喷水。请检查疏散是否合规。

**SKILL 应该怎么做**：

1. 识别类别：防火 ✓，建筑类型：办公（公共建筑），高度：18 层（高层）
2. 询问：
   > 我开始审查前需要确认：使用 GB 50016-2014 (2018 年修订版) 还是其他版本？
3. 用户确认后，加载 `references/fire-safety-checklist.md` 中相关条款。
4. 输出表格：

| 序号 | 条款 | 检查项 | 限值 | 设计值 | 判定 | 备注 |
|------|------|--------|------|--------|------|------|
| 1 | 第 5.5.8 条 | 每防火分区安全出口数 | 不少于 2 个 | 2 个 | ✅ 通过 | |
| 2 | 第 5.5.17 条 | 高层办公（一类）位于两个安全出口之间疏散门至最近出口距离 | 设有自喷时 ≤ 40m × 1.25 = 50m | 28m | ✅ 通过 | 自喷已设 |
| 3 | 第 5.3.1 条 | 一类高层公建防火分区最大面积 | 设自喷时 3000m² | 800m² | ✅ 通过 | |
| 4 | 第 5.5.13 条 | 高层公建首层疏散外门净宽 | 按计算确定，且不小于规范限值 | ⚠️ 用户未提供 | ⚠️ 需进一步信息 | 请提供首层疏散外门设计净宽 |

5. 末尾附完整免责声明。

**反例**（SKILL 不应该这样做）：

> ❌ "根据我的判断，整体疏散设计合规。"——没有逐项核对，不能给整体结论。
> ❌ "疏散距离 28 米 < 40 米，通过。" —— 没考虑公建类别（一类/二类）和是否设自喷的修正系数，限值不对。
> ❌ "建议增加一个安全出口。" —— 越权给设计建议；SKILL 的职责是判定合规性，不是替代设计师做方案。

## 与其他 SKILL 的边界

- 项目通信文档（变更单、纪要等）→ `aec-project-docs`
- 施工质量验收 → `aec-quality-acceptance`（如果有）
- 工地 HSE 检查 → `aec-hse-inspection`（如果有）
- 结构 / 节能 / 暖通的专业计算 → 各自的计算 SKILL（本 SKILL 不做计算，只做对照）

特别提醒：本 SKILL 不做**消防验收**或**消防设计审查**——那是政府审查的工作。本 SKILL 只做**自审清单**或**设计阶段的合规性预检查**。
