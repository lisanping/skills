# Word 输出样式规范

> **何时读取此文件**：当用户明确要求生成 .docx 文件，或要求"导出 Word 版本"、"出正式文件"时。

## 一、为什么需要统一样式

项目通信文档会被多方传阅、归档、打印。样式不统一带来的问题：

- 一份变更单字体五花八门，看起来不专业，甲方/监理印象分降低
- 表格宽度混乱，打印时跨页错位
- 颜色刺眼或太淡，黑白打印时关键信息丢失
- 不同 SKILL 输出的文档"长得不像同一家公司出的"

本文件规定 SKILL 生成 .docx 时的**默认样式**。如项目自有模板，应优先使用项目模板，覆盖本规范。

---

## 二、页面规范

### 纸张与边距

| 项目 | 默认值 | 说明 |
|------|-------|------|
| 纸张 | A4（21.0 × 29.7 cm） | 中国大陆默认；外资项目可用 US Letter |
| 上边距 | 2.54 cm（1 英寸） | |
| 下边距 | 2.54 cm | |
| 左边距 | 2.54 cm | 部分单位用 3.18 cm 以便装订 |
| 右边距 | 2.54 cm | |
| 装订线 | 0 cm | 仅在用户明确要求装订时设 0.5 cm |

施工日报建议横向（A4 横向），因表格列多，纵向排版列宽不够。

### 页眉 / 页脚

**页眉**（右对齐，字号小，灰色 #6B7280）：
```
{项目名称} | {文档类型}                              第 X 页 / 共 Y 页
```

**页脚**（居中，字号小）：
```
文档编号：{编号}        签发日期：{日期}        机密等级：{内部/项目内/公开}
```

页眉页脚字号建议 **9 磅**，颜色用次要灰，避免与正文抢视线。

## 三、字体

### 中文字体

| 用途 | 字体 | 字号 | 字重 |
|------|------|------|------|
| 大标题（文档名） | 微软雅黑 / 思源黑体 | 22 磅 | 粗体 |
| 一级标题（议题） | 微软雅黑 / 思源黑体 | 16 磅 | 粗体 |
| 二级标题 | 微软雅黑 / 思源黑体 | 13 磅 | 粗体 |
| 正文 | 宋体 / 思源宋体 | 11 磅 | 常规 |
| 表格内 | 宋体 / 思源宋体 | 10 磅 | 常规 |
| 页眉页脚 | 微软雅黑 | 9 磅 | 常规 |

### 英文字体

中英混排时，英文字体单独设置（中文用宋体时英文不要也用宋体，会显得拥挤）：

| 用途 | 字体 |
|------|------|
| 标题英文 | Arial / Calibri |
| 正文英文 | Times New Roman / Calibri |

### 字体选择的原则

- 中文标题用**黑体类**（微软雅黑、思源黑体），笔画粗细均匀，标题感强
- 中文正文用**宋体类**（宋体、思源宋体），笔画粗细对比，长文阅读舒服
- **不要**正文也用黑体——黑体长文阅读易疲劳
- **不要**用楷体、仿宋作正文——这两种是公文专用，项目通信文档不合适
- **不要**用艺术字体——任何花哨字体都不合适

## 四、颜色

### 色板

```
深蓝（标题、表头）        #1E3A5F
中蓝（次级强调）          #3B6CA8
浅蓝（表格背景）          #E8F1F9
橙（警示、重点）          #E89D43
深橙（必填提示）          #B45309
浅橙（警示背景）          #FEF3E7
红（不通过、否决）        #B91C1C
绿（通过、确认）          #1F7A3A
深灰（次要文字）          #374151
中灰（页脚、辅助）        #6B7280
浅灰（分隔线、底纹）      #E5E7EB
```

### 使用原则

- **正文文字**：黑色（#000000）或近黑（#1F2937）
- **标题文字**：深蓝（#1E3A5F），不要全用纯黑——纯黑标题缺乏层次
- **强调**：粗体优先于变色。需要变色时用深蓝或深橙，不要用艳红艳绿
- **彩色背景上的文字**：白色或与背景明显对比的色，确保黑白打印仍可读
- **避免**：纯红正文（除"不通过"等明确警示外）、霓虹色、纯白底配浅灰文字

### 黑白打印兼容

项目文档常被打印归档，且打印机往往是黑白的。检查方法：

- 用浅蓝 #E8F1F9 作表格背景：黑白打印为浅灰，不影响阅读 ✅
- 用红字标注关键信息：黑白打印为深灰，仍可见但失去警示性 ⚠️
- 用红色作整段文字：黑白打印后看起来和黑字无差别，但屏幕看刺眼 ❌

**不要把任何关键信息只通过颜色传达**——必须配合粗体、加框、文字标注。

## 五、表格

### 表格基础规则

- 全表使用**单线边框**，颜色 `#CCCCCC`，粗细 0.5–0.75 磅
- 表头行**填充背景色** `#1E3A5F`，文字白色粗体
- 数据行**交替填色**（`#FFFFFF` 与 `#F9FAFB`）便于阅读，可选
- 单元格**内边距**：上下 0.08 cm，左右 0.12 cm
- 单元格**对齐**：文字左对齐，数字右对齐，居中标识列居中

### 列宽建议

A4 纵向页面的可用宽度约为 16 cm。常见表格列宽：

| 列类型 | 建议宽度 |
|--------|---------|
| 序号 | 1.0–1.5 cm |
| 日期 | 2.5–3.0 cm |
| 短文字（姓名、编号） | 2.5–3.0 cm |
| 中等文字（描述） | 4.0–6.0 cm |
| 长文字（详细说明） | 余量 |

总宽度应 = 页面可用宽度，不要溢出（会被 Word 自动压缩，难看）。

### 表格的几条禁忌

- **不要把表格当排版工具**——比如用 1×2 表格做"左标题右日期"的页眉。这种用途用制表位（Tab Stop）实现。
- **不要用合并单元格做美化**——尤其不要为视觉效果合并表头单元格。合并破坏数据结构，使后续摘录困难。
- **不要用空表格作分隔线**——一行高度 0 的空表格只是看起来像分隔线，实际打印和导出 PDF 时常出问题。要分隔用段落底边框。
- **不要让表格跨页时表头丢失**——必须设置"在每页顶端以标题行形式重复出现"。

## 六、签字栏

签字栏是项目通信文档的**法律核心**。SKILL 生成时遵循以下硬性规则：

### 不要做的事

- ❌ 不要把任何人的名字预填进签字栏（即使用户提供了名字）
- ❌ 不要用 `_____________` 字符模拟签字线（打印效果差，且对齐困难）
- ❌ 不要在签字栏写"已签字"、"在此签字"等文字
- ❌ 不要用"※"、"☑" 等符号表示签字状态

### 推荐做法

签字栏用**表格 + 空白单元格 + 单元格底边框加粗**实现：

| 设计编制 | 设计审核 | 设计审定 |
|---------|---------|---------|
| _（空白，高度 1.5 cm，单元格底边框 1.5 磅）_ | _（同左）_ | _（同左）_ |
| 日期：_（空白，底边框 0.75 磅）_ | 日期：_（同左）_ | 日期：_（同左）_ |

签字单元格高度建议 **1.5 cm**——给手写签名留出足够空间。

### 电子签名

如项目使用电子签名，SKILL 不应自动嵌入任何人的电子签名图像。这必须由签字本人在自己的签字工具中操作。

## 七、列表

### 项目符号

- 一级用 `•`（U+2022 圆点），不要用 `●`（实心圆，太重）
- 二级用 `◦`（U+25E6 空心小圆）
- 三级用 `–`（U+2013 半角连字符）

### 编号列表

- 中文文档优先用 **"一、二、三、"** 作大标题，**"1. 2. 3."** 作小标题
- 不要混用 "1、2、3、" 和 "1. 2. 3." 风格——同一份文档中保持一致
- 多层嵌套使用 `1.1`、`1.2`、`1.1.1` 形式

### 列表的间距

- 列表项之间空 **0.2–0.4 行**，不要拥挤也不要过松
- 列表与前后段落之间空 **0.5–0.8 行**

---

## 八、段落

### 行距

- 正文：**1.5 倍行距** 或 **22 磅固定行距**
- 表格内：**单倍行距** 或 **18 磅固定行距**
- 标题：**单倍行距**

### 段间距

- 段前：**0.5 行**
- 段后：**0.3 行**
- 一级标题段前：**1.0 行**
- 标题不要使用首行缩进

### 首行缩进

- 中文正文：**首行缩进 2 字符**
- 英文正文：通常**不缩进**，段间空行

中英文混排时按主体语言的规则。

## 九、SKILL 实施要点

### 优先使用 docx-js（如可用）

```javascript
const doc = new Document({
  styles: {
    default: { document: { run: { font: { ascii: 'Calibri', eastAsia: '宋体', cs: 'Calibri' }, size: 22 } } },
    paragraphStyles: [
      { id: 'Heading1', name: 'Heading 1', basedOn: 'Normal', next: 'Normal', quickFormat: true,
        run: { size: 32, bold: true, color: '1E3A5F',
               font: { ascii: 'Arial', eastAsia: '微软雅黑' } },
        paragraph: { spacing: { before: 360, after: 180 }, outlineLevel: 0 } },
      // ... 其他样式
    ],
  },
  // ...
});
```

注意：

- `font` 字段必须分别指定 `ascii` / `eastAsia`，否则中文字体可能不生效
- 字号在 docx-js 中是 **半磅单位**（11 磅 = 22）
- 颜色不带 `#`，6 位 hex（如 `1E3A5F`）

### 验证

生成的 .docx 应通过验证：

```bash
python /mnt/skills/public/docx/scripts/office/validate.py output.docx
```

如验证失败，常见原因：

- 表格 `width` 与 `columnWidths` 之和不等
- 图片 `ImageRun` 缺 `type` 参数
- 标题样式 ID 拼错（必须是 `Heading1` 而非 `heading1`）

### 提供项目模板的情况

如用户上传了项目自有的 .docx 模板，SKILL 应：

1. **不要从零生成** —— 应该读取模板，提取样式，用模板的样式覆盖本规范的默认值
2. **保留模板的页眉页脚**（含 logo、项目名等）
3. **不要修改模板自带的样式定义**

## 十、最小可接受样式

如果以上所有内容都太复杂，**底线版本**是：

| 项 | 底线值 |
|----|-------|
| 字体 | 中文宋体、英文 Calibri |
| 字号 | 正文 11 磅 |
| 行距 | 1.5 倍 |
| 边距 | 上下左右各 2.54 cm |
| 表格边框 | 黑色单线 0.5 磅 |
| 表头底色 | 浅灰 #E5E7EB |
| 颜色 | 仅黑白 |

这套底线足以让文档看起来"不像草稿"，可以作为 SKILL 在用户没提任何样式偏好时的兜底配置。
