# RFI（信息请求单 / 工程询问单）模板

> **何时读取此文件**：当需要生成 RFI、Engineering Query、信息请求单、工程询问单时。

## 一、文档定位

RFI（Request For Information）是施工方/总包向设计方或业主提出技术问题、要求澄清的正式书面文件。RFI 的核心价值是**留痕**——所有问答都进入项目档案，作为后续争议的依据。

因此 RFI 必须满足：
- **问题陈述具体**——能让答复方知道要回答什么
- **答复期限明确**——避免被无限期拖延
- **可追溯**——编号、日期、引用图纸/规范都要齐全

## 四、字段说明

### 必填字段（缺一不可）

| 字段 | 为什么必填 |
|------|-----------|
| RFI 编号 | 留痕检索的唯一键 |
| 签发日期 | 计算答复期限的起点 |
| 答复期限 | **缺失会导致无限期拖延**——这是 RFI 最常见的失败模式 |
| 提资人 + 联系方式 | 答复方有问题时能联系上 |
| 涉及图纸编号 + 版次 | 没有版次的图纸引用是无效引用——图纸已升版的情况下问题可能已不存在 |
| 问题描述 | 不能只写"图纸有问题，请处理" |
| 分发列表 | 决定谁会收到这份 RFI |

### 答复期限的设定原则

- 一般技术澄清：5 个工作日
- 涉及方案调整、需要内部协调：7–10 个工作日
- 紧急（影响在建工序）：明确标注"紧急"+ 具体小时数（如 48 小时内）
- **不要写"尽快"、"请尽早"**——这等于没设期限

### 问题描述的写法

好的问题描述包含三段：

1. **当前情况**（事实陈述，不带情绪）
2. **疑点**（为什么这是个问题，会引起什么后果）
3. **决策需求**（需要对方做出什么决定/给什么信息）

反例：
> ❌ "图纸标注有问题，影响施工，请尽快处理。"

正例（见上方示例的"问题描述"段落）。

### 提资单位建议方案

虽然标注为可选，但**强烈建议提供**。理由：

- 答复方更容易在你的方案上"打勾"或"打叉"，效率高
- 即使被否决，也能引出"为什么不行"的反馈，比开放式问题信息量大
- 留痕显示提资方做过功课，不是甩问题

## 五、常见错误

1. **答复期限缺失或模糊**——前面已强调，这是头号问题。
2. **图纸引用没写版次**——`S-21-301` vs `S-21-301 Rev.B` 是两份不同的图。
3. **一份 RFI 包含多个无关问题**——一事一单，便于追踪和归档。如果实在相关，可以在一份里分 1/2/3 列出，但每条都要单独标记答复状态。
4. **"问题描述"实际上是抱怨**——"设计图纸如此粗糙，让我们如何施工"——这种措辞会让答复方进入防御状态，得不到有用回复。坚持事实陈述。
5. **遗漏抄送给监理**——按合同惯例，监理应当知晓所有 RFI 流转。漏抄会导致后续争议中监理无法证明已知情。
6. **答复区由提资方预填内容**——答复区必须留空给答复方填写。提资方不能"代填"任何答复内容。
7. **附件清单写"详见附件"但不编号**——附件必须编号（附件 1、附件 2…），便于答复时定向引用。

## 六、与其他文档的衔接

- 如果 RFI 答复**引起设计变更**，答复方应同步出具《设计变更单》，并在 RFI 答复区填写变更单编号。
- 如果 RFI 答复**引起工程量变化**，总包应据此提交《工程量变更签证》。
- RFI 应与项目的 **RFI Log（信息请求台账）** 同步——SKILL 在生成 RFI 时建议同时更新台账行（如用户提供）。
