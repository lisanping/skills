# Testing and evaluation methodology

SKILLs in this repository fall into two categories, each tested differently:

| Category              | Examples                                       | Test method                                                |
| --------------------- | ---------------------------------------------- | ---------------------------------------------------------- |
| **Code-oriented**     | `aec-building` BREP engine                     | `pytest` unit tests                                        |
| **Document-oriented** | `aec-compliance-checklist`, `aec-project-docs` | **Prompt-based eval** (manual run + JSON binary judgement) |

---

## 1. Code-oriented SKILLs (pytest)

Live under `packs/<pack>/tests/test_*.py`. Conventions:

```bash
cd packs/<pack>
conda activate <pack>
pytest tests/ -v
```

- Heavy-dependency tests (e.g. OCCT / CadQuery) are tagged with a keyword so they can be skipped: `pytest -k "not BREP"`.
- Fixtures live under `tests/fixtures/` (small samples are tracked by git; large samples go through git-lfs or are generated by a script).

---

## 2. Document-oriented SKILLs (prompt-eval)

### File layout

```text
packs/<pack>/tests/
├── BASELINE-v1.0.md              # baseline manifest + change-control rules
├── test-design-rationale.md      # how the test set was designed
├── <skill>.evals.json            # cases + expectations
├── cross-skill.evals.json        # cross-SKILL collaboration cases
└── results/                      # one report per run (never overwritten)
    ├── 2026-04-29-baseline.md
    └── 2026-05-03-fix-rfi-template.md
```

### `evals.json` format

```json
{
  "skill": "aec-project-docs",
  "version": "1.0",
  "cases": [
    {
      "id": "rfi-001",
      "input": "Draft an RFI about a thickness change to the level-3 floor slab.",
      "expectations": [
        "Includes the RFI number field",
        "Includes originator / recipient fields",
        "Includes a technical question section",
        "Format follows references/rfi-template.md"
      ]
    }
  ]
}
```

### Execution flow

1. The author runs `input` through the SKILL workflow in a real agent.
2. They mark each `expectation` as ✓/✗ (binary judgement).
3. The report is written to `tests/results/<date>-<topic>.md`, recording the pass rate and any failure causes.
4. Compare the pass rate against the baseline; if it dropped, fix the SKILL or update the test before merging.

---

## 3. Baselines

Each pack maintains a `tests/BASELINE-vX.Y.md` containing:

- The test-set manifest (file path + case count + expectation count)
- The current pass rate
- The matching git tag (e.g. `aec-skills-baseline-v1.0`)
- The last update date

### When to bump the baseline

- A new test set is added.
- The pass rate improves (and not by relaxing test cases).
- A SKILL undergoes a major version bump.

When bumping, create a new tag of the form `<pack>-baseline-v<major>.<minor>`.

---

## 4. SOP for modifying a SKILL

```text
1. Before changing: re-run the affected test set against the current baseline; record the pass rate.
2. One change at a time: SKILL.md / references / scripts / evals — only one of them.
3. Re-run the full suite: pass rate must be ≥ the baseline before merging.
4. If the pass rate dropped: decide whether the SKILL regressed or the test cases need updating.
5. For any change: append a new report under tests/results/ (never overwrite).
```

---

## 5. CI

Repository-level CI ([.github/workflows/](../.github/workflows/)) only runs:

- `scripts/lint_skill.py` — frontmatter validation
- `scripts/check_naming.py` — naming validation

**Neither pytest nor prompt-eval runs in CI.** The former is too heavy per pack (OCCT and similar dependencies); the latter is manual by design. Pack maintainers run them locally and commit the results into `tests/results/`.
